# 08 — Frontend & SEO

> 🧭 **Status:** 📝 Spec · **Tier:** 🏆 Best-in-class (design) → **Target: 🏆 Best-in-class** · **Phase 1** · live tracker: [STATUS.md](./STATUS.md) <!--STATUS-BADGE-->

> **Context:** One Next.js app serves both the public, SEO-driven surface (landing, stock pages, blog, public track record, backtest tool) and the private authenticated dashboard (picks, paper trading, journal, analytics). This doc defines routes, rendering strategy, the SEO engine (the free-user acquisition channel), and the dashboard UX that delivers Layers 1–2 (survival + process).

## 1. Routes & rendering

| Path | Purpose | Auth | Render | Notes |
|---|---|---|---|---|
| `/` | Marketing landing | Public | SSR | Value prop = process + transparency, NOT "guaranteed profit" |
| `/pricing` | Plans | Public | SSG | Free / Pro ₹499 / Premium ₹999 |
| `/about` | About / mission | Public | SSG | |
| `/stocks/[symbol]` | Auto AI analysis per NSE stock | Public | **ISR** (revalidate daily) | SEO engine — hundreds of pages |
| `/sectors/[sector]` | Sector analysis | Public | ISR | SEO |
| `/backtest` | Public "prove it" tool | Public | SSR | Reads cached `backtest_runs` |
| `/track-record` | Honest public scorecard | Public | ISR | The trust weapon — R-based, net, all trades |
| `/blog/[slug]` | SEO articles (weekly AI-generated) | Public | ISR/SSG | Long-tail keywords |
| `/login`, `/signup` | Auth | Public | Client | Supabase Auth |
| `/dashboard` | Picks, paper trades, P&L, journal | 🔒 | Client + protected | Layers 1–2 live here |
| `/dashboard/analytics` | Personal expectancy & discipline | 🔒 | Client | |
| `/admin` | Management panel | 🔒 Admin | Client | `is_admin` only |

**`middleware.ts`:** intercepts protected routes; unauthenticated → redirect to `/login` **preserving the callback URL** so the user returns to where they were.

**Rendering rule of thumb:** anything that needs Google = SSR/ISR (public). Anything personal/private = client-side + protected (no SEO value, must not be indexed; add `noindex`).

**Locale prefix:** every route is under `/[locale]` (`/en/…`, `/hi/…`) — i18n via next-intl (doc 15). Middleware resolves locale (user pref → admin default → header → `en`) alongside auth + RBAC + block-status.

**Admin routes** (`/[locale]/admin/*`) are RBAC-gated **full-page** screens (appearance, integrations, users, sessions, analytics, audit, roles, settings) — see docs 16–20. **Never dialogs** for substantial flows; a back button returns to the list. Modals only for short confirmations.

### 1b. Theming, i18n & animation integration (docs 14, 15)
- **Providers** wrap the app: `ThemeProvider` (next-themes, CSS-variable tokens) + `IntlProvider` (next-intl). The resolved theme/font/locale = **user override → platform default (admin) → safe fallback** (docs 14, 16); applied pre-hydration to avoid flash.
- **No hardcoding:** all strings via `t()` (doc 15), all colors via tokens (doc 14), all settings from the backend. Loading/empty/error states are real (data is API-driven).
- **Locale + theme switchers** in user settings (full page); admin sets platform defaults + locks (doc 16).
- **Framer Motion** for purposeful motion within the performance budget (doc 14 §4, §7); respect `prefers-reduced-motion`.
- **Responsive + mobile-first from day one** (doc 14 §6): dense admin tables → cards on mobile; bottom-nav/drawer on mobile, sidebar on desktop.

## 2. The SEO engine (free-user acquisition)

SEO is the long-term free-traffic channel. Honest caveat: it's a **12–18 month game** and won't carry launch — near-term distribution is content + the public track record on YouTube/Twitter/Reddit (r/IndianStockMarket). But built right, it compounds.

**Programmatic pages (the scale play):**
- `/stocks/[symbol]` for **every** NSE stock, ISR-revalidated daily from `ai_picks`/`stock_meta`. Hundreds of long-tail pages ("HAL share analysis", "SBI swing trade setup") from day one.
- `/sectors/[sector]` for each sector.
- Each page: latest technical analysis, indicator snapshot, regime context, educational framing, and a CTA to the dashboard.

**Content pages:**
- Weekly AI-generated `/blog/*` ("Top 5 swing setups this week — June 2026"), generated by the `seo_content` cron (doc 07).
- Educational guides ("How to swing trade SBI — with risk management") — these rank *and* reinforce the discipline brand.

**Dynamic, admin-controlled marketing (doc 21):**
- Marketing pages (landing, feature, pricing, about, + admin-created pages) are **DB-driven but rendered server-side (ISR)** via a `BlockRenderer` that maps typed content blocks → themed/animated components. Content is **seeded on first boot**, then fully editable by Super Admin (no hardcoded copy).
- **`generateMetadata()` reads `seo_meta` per page** server-side (title, description, canonical, OG/Twitter, JSON-LD, robots) — all editable from the admin panel.
- **On-demand revalidation** when admin publishes → live in seconds, no redeploy. Drafts preview via a token-gated `noindex` route.
- Blocks are responsive + dark/light/system + animated by construction, so admin edits content, not styling — the site can't break or go off-brand.

**Technical SEO:**
- **JSON-LD structured data** on every page (Article, FinancialProduct, FAQ, Organization, BreadcrumbList) — **editable per page from admin** (doc 21) — so Google **and** AI search (ChatGPT/Perplexity/Claude) can parse and cite; especially the honest track record. Optional **`llms.txt`** for AI ingestion.
- Clean semantic HTML, fast Core Web Vitals (Next.js SSR/ISR + Vercel edge), sitemap.xml, robots.txt, canonical URLs, OG images.
- **Compliance in copy:** every public analysis page carries the "educational analysis, not advice" framing and avoids imperative buy/sell language (doc 09).

## 3. Dashboard UX — where Layers 1–2 live (the actual product)

The dashboard is not "a list of picks." It's the **risk + discipline** surface.

**Daily picks view (Layer 3, the hook):**
- Regime banner up top ("Market is BULLISH — we're looking for longs" / "BEARISH — sitting in cash, here's why"). Cash days are framed as a *win* (capital preserved), with educational content.
- Each pick card: **score with breakdown** (RS 22/25…), knockouts passed, full trade plan (entry/stop/T1/T2/R:R), **position size computed from the user's own capital**, portfolio-heat impact, and the Hinglish explanation.
- One-tap **"Paper trade this"** → pre-fills the risk-sized order.

**Risk engine (Layer 1, the moat):**
- Position-size calculator front and center — user can't enter a size that breaches risk %, per-stock cap, or portfolio heat (enforced; the UI shows *why* if blocked).
- Live **portfolio heat meter** (total risk across open positions).

**Paper trading + journal (Layer 2, retention):**
- Open positions with live P&L and current stop (incl. breakeven moves).
- **Auto journal:** entry reason, planned R, outcome — logged automatically, annotatable.
- **Post-trade review:** surfaced insights ("you exited 7 of your last 10 winners before T1", "your discipline score dropped this week").

**Analytics (Layer 2):**
- **Expectancy (avg R) as the headline**, not win rate. Profit factor, max drawdown, avg hold, best sector, discipline score, equity curve (Recharts).

**Charts:** TradingView `lightweight-charts` for price/candles; Recharts for P&L/equity curves.

## 4. Components & state

- **shadcn/ui + Tailwind** for all UI primitives.
- **TanStack Query** for all server data (picks, portfolio, analytics) — caching + background refresh.
- **Zustand** for light client state (UI prefs, calculator inputs).
- **API client** in `lib/` typed from the backend OpenAPI schema (`packages/shared-types`) so frontend/backend never drift.
- **Supabase client** for auth/session; session checked in `middleware.ts` and per protected page.

---

## 5. Performance & accessibility

- ISR for all public data pages → fast + fresh + cheap.
- Image optimization (next/image), font optimization, route-level code splitting.
- Mobile-first (85% of Indian traders are on Android phones) — dashboard must be excellent on mobile web before the native app exists (doc 13).
- Accessible color/contrast for P&L (don't rely on red/green alone).
